# Changelog

## [1.10.6] - 2026-03-21

### Changed
- 主动发送结果判定收敛为 `send_message_to_user` 返回包含 `Message sent to session` 才视为成功，避免误判。
- 主动发送新增消息对象兜底：构造带 `chain` 属性的消息结果对象传给 `context.send_message`，兼容不接受纯文本/列表的运行环境。
- 发送失败时保留最后一次真实错误，不再被统一文案覆盖，提升诊断可读性。

## [1.10.5] - 2026-03-21

### Changed
- 主动发送参数收敛为 AstrBot 推荐契约：优先 `messages + unified_msg_origin`，其次 `messages + session_id(纯ID)`，最后 `messages`。
- 发送结果新增成功判定，避免工具返回失败文本时被误判为发送成功。
- 移除已知不兼容的 `[Plain]` 列表发送兜底，避免触发 `'list' object has no attribute 'chain'`。

## [1.10.4] - 2026-03-21

### Changed
- 主动发送新增 `context.send_message_to_user` 直连尝试，优先按多种参数形态发送，减少仅依赖工具执行器导致的失败。
- 发送链路记录最近一次错误原因，并在 `提醒诊断/sub debug` 输出，便于直接定位失败参数契约。

## [1.10.3] - 2026-03-20

### Changed
- 主动发送新增工具链路优先尝试：通过 `send_message_to_user` 工具发送提醒，兼容 `unified_msg_origin/session_id/umo/session` 多种参数形态。
- 主动发送失败时回退 `context.send_message`，保留原有发送兼容路径。
- 新增提醒命令入口日志，便于确认 `提醒诊断/sub` 是否已被 TodoPal 插件命中处理。

## [1.10.2] - 2026-03-20

### Added
- 新增提醒诊断命令：`提醒诊断` 与 `sub debug/test`，可即时输出提醒触发条件与主动发送链路可用性。

### Changed
- 增强主动发送链路日志：`_send_text_to_origin` 记录各调用形态失败原因，不再静默吞错。
- 本地循环提醒在尝试发送但失败时输出告警日志，便于定位“定时已命中但未发出”场景。
- 提醒无待办时补充调试日志，区分“未命中发送条件”和“发送失败”。

## [1.10.1] - 2026-03-20

### Added
- 新增固定订阅制提醒能力：支持 `sub on/off/list` 与 `订阅提醒/取消提醒/提醒订阅` 命令管理个人提醒订阅状态。
- 新增配置项 `reminder_require_subscription` 与 `reminder_subscription_default_on`，可切换“仅订阅用户推送”或“全量用户推送”。

### Changed
- 本地循环提醒发送条件增加订阅状态判断，默认仅向已订阅用户推送。

## [1.10.0] - 2026-03-20

### Changed
- 提醒默认改为插件本地循环调度：`use_system_scheduler_for_reminder` 默认值调整为 `false`，避免依赖主系统 future task。
- 新增提醒文案模式配置：`reminder_text_mode`，默认 `template`，可切换为 `llm`。
- 新增提醒模板配置：`reminder_template`，支持 `{pending_count}`、`{pending_preview}`、`{top1}`、`{top2}`、`{top3}` 占位符。
- 主动提醒发送链路支持纯模板直出，无需 LLM 也可稳定推送。

## [1.9.2] - 2026-03-19

### Fixed
- 修复部分环境下 `Context` 未直接暴露 `create_future_task/delete_future_task/list_future_tasks` 时无法创建循环任务的问题。
- 新增工具执行器兼容层：可通过 `call_tool/invoke_tool/execute_tool/run_tool/call_func_tool` 代理调用系统定时任务工具。
- 新增调度模式启动日志，明确当前是 `system` 还是 `local` 调度路径，便于快速排查“没有循环任务”问题。

## [1.9.1] - 2026-03-19

### Fixed
- 修复用户上下文注册会覆盖提醒任务字段的问题，`register_user` 改为增量合并，避免 `reminder_task_id` 在每次对话后被清空。
- 修复主系统调度判定过早导致本地提醒被误停的问题：仅当用户已成功创建系统提醒任务时才停用本地提醒分支。
- 修复 future task 返回结构兼容性不足问题：补充嵌套 `task/data/result` 的任务 ID 与名称解析。
- 修复任务创建易失败但不可观测问题：增加创建尝试日志、返回类型日志与同步结果日志，便于定位参数契约差异。

### Changed
- 调整 `future_task` 创建参数序列，优先尝试 `task_type=active_agent`。
- 优化 reminder note 指令模板，明确要求执行时使用 `send_message_to_user` 发送提醒消息。

## [1.9.0] - 2026-03-19

### Added
- 新增提醒调度适配层：支持优先接入主系统 `future_task`（`create_future_task / delete_future_task / list_future_tasks`）托管循环提醒。
- 新增提醒任务幂等同步机制：按用户维度维护 `reminder_task_id/reminder_task_name/reminder_signature`，配置变更自动重建任务。
- 新增 `use_system_scheduler_for_reminder` 配置项，可在 WebUI 切换“主系统调度”与“插件本地循环”。

### Changed
- 命令与工具入口在注册用户上下文后会同步提醒任务，降低“配置改了但任务未更新”问题。
- 当主系统调度可用时，本地 `cron` 不再执行提醒发送，仅保留本地兜底与其他后台能力。

## [1.8.2] - 2026-03-19

### Changed
- 主动提醒与主动总结新增发送兼容层，统一尝试多种 `send_message` 参数形态，降低因接口签名差异导致的“未发送”问题。
- 主动提醒与主动总结新增模板兜底：当 LLM 不可用或生成失败时，仍发送纯文本提醒/总结，避免整条链路被 provider 阻断。
- 主动提醒与主动总结新增错误日志上下文（携带用户维度），便于快速定位具体失败用户。

## [1.8.1] - 2026-03-19

### Changed
- 主动提醒/总结链路增加运行时 provider 刷新：用户缓存无 provider 时，会基于 origin 重新解析并回写，减少“有待办但不提醒”场景。
- LLM Tool 路径（`todo_check/todo_add/todo_done/todo_fix/todo_delete`）补齐用户上下文注册，确保工具调用也能刷新 `origin/provider_id`。
- `cron` 用户扫描增加空字段保护，避免异常用户记录影响整轮提醒任务执行。
- 主动提醒与总结在 provider 不可用时输出调试日志，便于快速定位未发送原因。

## [1.8.0] - 2026-03-18

### Added
- 新增删除能力：支持 `del`、`delete`、`rm` 显式命令按序号或关键词删除待办。
- 新增 `todo_delete` LLM Tool，支持主系统在自然对话中调用删除能力。
- 新增删除服务与存储实现，统一返回 `deleted_count`、`deleted_items` 与标准 `message`。

### Changed
- 扩展 `todo` 智能意图，新增 `delete` 类型解析与执行分支。

## [1.7.1] - 2026-03-18

### Changed
- 统一 `add/done/fix/check` 服务层返回协议，新增标准化 `message` 字段，命令路径与工具路径共享同一套结果语义。
- `todo` 与 `add` 的新增链路统一复用 `_service_add`，支持“仅解析预览（persist=False）+ 用户确认后保存”的一致交互。
- `done` 与 `fix` 命令输出改为优先消费服务层结果文案，减少分散硬编码提示，提升对话一致性。
- 清理历史遗留分支 `todo_old_overwrite_mode_disabled`，收敛到当前有效主流程（append + confirm）。

## [1.7.0] - 2026-03-18

### Added
- 新增 `todo_check`、`todo_add`、`todo_done`、`todo_fix` 四个 LLM Tool 接口，支持主系统在自然对话中直接调用待办能力。
- 新增能力服务层：统一 `check/add/done/fix` 的结构化返回结果，便于主系统按人格与上下文组织最终回复。

### Changed
- 消息入口从全量拦截改为仅处理显式命令前缀（`todo|add|done|fix|check`），避免与系统默认对话链路冲突。
- 保留显式命令兼容路径，同时将核心数据能力下沉到可复用服务函数，减少“插件自成对话系统”的割裂感。

## [1.6.1] - 2026-03-17

### Added
- `check` 场景新增“先回应再清单”的双阶段输出流程：先发送人格化短回应，再发送结构化待办列表。
- 新增流式回复尝试通道：优先使用流式 LLM 输出生成前置回应，失败时自动回退为非流式。

### Changed
- 调整 `check` 展示逻辑，避免把开场语和清单拼成单条长消息，降低“先反问后列清单”的不自然感。

## [1.6.0] - 2026-03-17

### Added
- `check` 查询新增日期语义支持：可识别今天/明天/后天/明确日期（`YYYY-MM-DD`、`YYYY/MM/DD`、`3月18日`）。
- 新增人格化前缀回复通道：在清单展示场景使用“人格开场 + 结构化正文”的组合，提升人格生效可观测性。
- 新增 JSON 自动修复重试机制：当 LLM 首次输出非合法 JSON 时，自动发起修复请求并再次解析。

### Changed
- 优化新增待办确认前的回复文案：改为先给出“拟执行计划提示”，再展示结构化预览并等待确认。
- 升级智能意图协议中 `check` 的 payload 约定，支持 `{"date":"YYYY-MM-DD"}` 或 `null`。

## [1.5.0] - 2026-03-16

### Added
- **可视化配置面板**：新增 `_conf_schema.json`，支持在 AstrBot WebUI 中直接调整插件行为。
- **自定义特征词库**：现在可以在配置页中随意添加/删除触发插件的自然语言关键词。
- **Persona 人格设定**：支持在配置页选择已有的系统人格，助手主动发消息时将自动使用该人格语气。
- **智能循环提醒**：支持设置“工作时间段”和“提醒间隔”，只要有未完成的待办，助手会在指定时间内定期催促你。
- **晚间总结报告**：支持设定固定时间（如每天 23:00），助手会根据今日完成情况生成总结报告。
- **跨日自动顺延**：新增后台任务机制，如果开启配置，每天 0 点（或首次访问时）会自动将昨天未完成的任务顺延至今天。

## [1.4.0] - 2026-03-16

### Added
- **智能意图识别 (Smart Agent Mode)**：升级 `todo` 指令为智能入口。现在你可以直接说 `todo 把第二条改成开会` 或 `todo 买菜做完了`，插件会自动识别并执行 `fix` 或 `done` 操作，无需记忆特定指令前缀。
- 支持 `todo` 后接纯自然语言，如 `todo 看看还有什么` (Check) 或 `todo 加上买牛奶` (Add)。

### Changed
- `todo` 指令不再默认覆盖当日清单，而是默认为**智能追加/处理**模式。为了防止数据误删，现在 `todo` 和 `add` 都会追加待办事项。如果需要覆盖，请手动删除旧数据（覆盖功能将在后续版本重新设计）。
- 保持 `add`, `done`, `fix`, `check` 作为快捷指令，供需要精准控制或不希望消耗 LLM Token 的场景使用。

## [1.3.0] - 2026-03-16

### Added
- 新增 `fix` 指令：支持修改已存在的待办内容（如 `fix 2 改成开会`），解决了之前无法修改只能覆盖的问题。
- 新增 `check` 指令：支持随时查看今日待办清单，无需触发解析流程。

### Changed
- 优化 `todo` 指令逻辑：现在作为“重置/新建”指令，如果检测到当日已有数据，会发出覆盖警告，防止误操作。

### Fixed
- 修复 `done`、`check`、`fix` 指令执行时可能出现的 `async_generator` 报错或无响应问题（重构了内部处理函数的返回机制）。

## [1.2.0] - 2026-03-16

### Changed
- 重构命令路由逻辑，引入明确的前缀机制：
  - `todo`：新建并覆盖当前清单。
  - `add`：在当前清单基础上追加新事项。
  - `done`：标记任务完成（支持批量，如 `done 1, 2` 或 `done 开会`）。
- 取消了原来依赖 LLM 意图识别的“追加/覆盖”提示，改为由指令前缀显式控制。
- 优化了待办事项的展示格式，已完成的项现在会直观地显示 ✅ 标记。
- 移除了 `done` 指令对 LLM 的依赖，直接在本地进行正则和内容匹配，大幅提升响应速度和准确率。

## [1.1.0] - 2026-03-16

### Added
- 实现本地 JSON 存储功能，支持按用户、日期归档待办事项。
- 新增待办事项交互确认流程（Wait-for-Confirmation）。
- 支持“追加”或“覆盖”今日已存在的待办事项。
- 新增自然语言标记完成功能（支持“第X个做完了”或“买菜做完了”）。

### Fixed
- 修复 `AiocqhttpMessageEvent` 等部分适配器缺少 `platform_name` 属性导致的崩溃问题。
- 优化 LLM 解析逻辑，增强对非标准 JSON 返回的鲁棒性。

## [1.0.0] - 2026-03-16

### Added
- 初始版本发布。
- 支持 `todo` 关键词解析待办事项。
- 实现基于 LLM 的自然语言理解，自动提取日期、时间和任务内容。
- 支持一次性添加多个待办事项。
- 智能时间推断（默认解析为当天）。

### Fixed
- 修复 LLM 返回 Markdown 代码块格式导致解析失败的问题。
- 修复部分 AstrBot 版本下无法获取 Provider ID 的问题（兼容 v4.5.7+ 新 API）。
